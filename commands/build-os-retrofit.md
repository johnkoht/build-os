# Build OS Retrofit

For projects where build-os was added after the project was already underway, or where the scaffolding has rotted. Audits the project against the build-os structure and proposes specific fixes.

## When to Use

- ✅ Project has `.build/expertise/{domain}/` dirs with only `.gitkeep` files
- ✅ Project has `.build/expertise/` but no domains identified yet
- ✅ Project predates build-os and was retrofitted without filling in profiles
- ✅ Profiles drifted so far they need a full re-pass
- ❌ Brand-new project — use `new-project` CLI instead
- ❌ Single profile needs refresh — just edit it directly, or let `/wrap` flag it

---

## Phase 1: Audit

Walk the project and produce a gap report. Check:

**Build OS scaffold**
- [ ] `.build/` exists with `expertise/` and `agents/` subdirs
- [ ] `.build/README.md` exists
- [ ] Each `.build/expertise/{domain}/` has a populated `PROFILE.md` (not `.gitkeep`)
- [ ] Each populated PROFILE.md has valid frontmatter (`domain`, `scope`, `last_validated`)

**Project memory**
- [ ] `plans/` exists
- [ ] `memory/MEMORY.md` exists
- [ ] `memory/entries/` exists

**CLAUDE.md wiring**
- [ ] Project `CLAUDE.md` references `~/.claude/build/AGENTS.md`
- [ ] Project `CLAUDE.md` has the slash-command workflow table
- [ ] Project `CLAUDE.md` declares `QUALITY_GATES`

For each ✗, classify severity:
- **blocker** — subagents can't function correctly (e.g., no CLAUDE.md AGENTS.md ref)
- **gap** — feature works but Layer 4 missing (e.g., empty expertise dirs)
- **migration** — exists but needs upgrade (e.g., PROFILE.md missing frontmatter)

---

## Phase 2: Domain Detection

Identify the project's natural domains. Prefer in order:

1. **Existing per-service CLAUDE.md files** — `services/*/CLAUDE.md`, `packages/*/CLAUDE.md`, `apps/*/CLAUDE.md`
2. **Top-level concern dirs** — `api/`, `mobile/`, `web/`, `worker/`
3. **Existing empty stubs** — `.build/expertise/{name}/` dirs without PROFILEs hint at domains the user already named
4. **Ask the user** — if none of the above are clear

Each detected domain gets a **source quality** rating:
- **strong** — CLAUDE.md exists at the domain root, >50 lines, has architecture detail
- **weak** — CLAUDE.md exists but is thin or generic
- **absent** — no domain-level docs

Source quality drives Phase 4 routing (strong → Path A, weak/absent → Path B).

## Phase 3: Propose

Output the gap table and proposed fixes. Get user confirmation before any write.

```markdown

|------|--------|----------|--------------|
| .build/ scaffold | ✅ exists | — | — |

| PROFILE.md frontmatter | N/A (no profiles yet) | — | — |
| memory/MEMORY.md | ✅ exists | — | — |
| CLAUDE.md → AGENTS.md ref | ✅ present | — | — |

### Domains detected

| Domain | Source | Quality | Path |
|--------|--------|---------|------|
```

If user says "adjust", iterate on the domain list before proceeding.

---

## Phase 4: Execute

For each domain, branch on source quality:

### Path A — strong source exists

The domain already has a substantive CLAUDE.md. Don't paraphrase it — that creates a sync burden. The PROFILE.md becomes a thin pointer that declares scope and defers to the source:

```markdown
---
domain: {domain}
scope:
  - services/{domain}/**
  - tests/test_{domain}*.py
last_validated: {today}
---

# {Domain} domain

> Generated by /build-os-retrofit on {today}. Review and refine.

**Source of truth:** [`services/{domain}/CLAUDE.md`](../../../services/{domain}/CLAUDE.md)

Subagents touching this domain MUST read that file. This profile exists to:
- Declare `scope` so `/wrap` knows when domain code changed
- Hold subagent-specific architectural notes that don't belong in the human-facing CLAUDE.md (add below as you discover them)

## Subagent-specific notes

_(Add architectural learnings here as `/post-mortem` routes them.)_
```

Scope globs come from the actual paths you found in Phase 2 — don't invent them.

### Path B — weak or absent source

No good source to defer to. Build the profile interactively, section by section. Follow the structure of `~/.claude/build/agents/`-flavored profiles (Stack, Architecture Map, Rules, Anti-Patterns, Required Reading).

Propose each section, get confirmation, then move to the next. Don't write a 200-line profile in one shot — that's how slop gets shipped.

```markdown
## Drafting {domain}/PROFILE.md

### Section 1 of 5: Stack

Based on package.json / pyproject.toml / etc.:

  {drafted stack section}

Looks right? (yes / edit / skip)
```

Continue for Architecture Map, Rules, Anti-Patterns, Required Reading.

### Migration: profiles without frontmatter

If a populated PROFILE.md exists but lacks `domain`/`scope`/`last_validated` frontmatter, prompt the user to choose scope globs (suggest reasonable defaults from the profile content) and set `last_validated` to today. Prepend the frontmatter; don't touch the existing body.

## Rules

- **Never overwrite** existing populated content. Migrations add frontmatter; they don't rewrite bodies.
- **Always show the diff** before writing. The user confirms each file.
- **Mark every generated file** with `> Generated by /build-os-retrofit on {date}. Review and refine.`
- **`last_validated` defaults to today** for new and migrated profiles.
- **Don't backfill LEARNINGS.md.** Those emerge from work, not retroactive invention.
- **Don't extract sections** from a domain's CLAUDE.md into its PROFILE.md (Path A). The pointer model means CLAUDE.md stays the single source of truth.

---

## Out of Scope

- **New-project bootstrapping** → use the `new-project` CLI
- **Single-profile refresh** → just edit it directly; bump `last_validated`
- **PRD/plan archeology** → only touches `.build/`, `memory/`, and CLAUDE.md wiring
- **LEARNINGS.md generation** → emerges from work, not retrofit

# Pre-Mortem: skill-model-config

## Risk 1: Frontmatter parser fragility on existing skills

**Problem**: Plan proposes a "simple line-based parser" for skill frontmatter, but `commands/hotfix.md` and `commands/build-os-retrofit.md` already use `triggers:` as a **YAML list** (multi-line, `- item` shape). A line-based `key: value` parser will corrupt these files on sync — either dropping the list or emitting broken YAML.

**Mitigation**: Use `pyyaml` (which we're already depending on) to parse *and* serialize frontmatter. Load with `yaml.safe_load`; dump with `yaml.safe_dump(..., sort_keys=False, default_flow_style=False)` to preserve field order and block style. Do not hand-roll.

**Verification**: After first sync, `commands/hotfix.md` and `commands/build-os-retrofit.md` still show `triggers:` as a YAML list identical in shape to before (spot-diff pre/post-sync).

## Risk 2: pyyaml install ergonomics on modern macOS

**Problem**: Plan says `install.sh` runs `python3 -m pip install --user pyyaml` if import fails. On macOS 14+ (PEP 668, "externally-managed-environment"), system Python refuses `pip install` without `--break-system-packages` or a venv, so the install step will error and block onboarding. This also changes `install.sh`'s ethos from pure-bash-symlink to opinionated-Python-installer.

**Mitigation**: Don't auto-install pyyaml in `install.sh`. Instead: `bin/build-config` detects missing pyyaml at first CLI use and prints a clear multi-line message with three options (`pipx install pyyaml`, `brew install libyaml && pip install --user pyyaml`, or a venv). `install.sh` stays symlink-only and prints a hint at the end: "Run `bin/build-config init --global` when ready to configure per-skill models — requires pyyaml." Users who don't touch model config never need pyyaml.

**Verification**: Fresh `install.sh` run on a machine without pyyaml completes without error. Running `bin/build-config init` without pyyaml exits with the actionable install message, not a Python traceback.

## Risk 3: Invalid model names silently break skills

**Problem**: User types `default_model: opuss` (typo) or `default_model: gpt-4` (wrong provider). Sync happily writes `model: opuss` to every skill file. Next Claude Code session errors or silently ignores unknown model — user hits a confusing failure with no signal from the tool that inserted the bad value.

**Mitigation**: `bin/build-config` keeps a known-good allowlist (`opus`, `sonnet`, `haiku`, `inherit`) and warns on any value outside it with a "did you mean …?" hint. Doesn't hard-error — Claude Code accepts long-form model IDs and future values, so unknown ≠ invalid. Sync exits non-zero on warnings unless `--force` is passed, so CI-ish workflows notice.

**Verification**: `bin/build-config sync --global` with `default_model: opuss` prints `warning: unknown model "opuss" — did you mean "opus"?` and exits non-zero. With `default_model: opus`, no warning.

## Risk 4: Managed-file marker safety for hand-edits

**Problem**: The `managed: build-os` frontmatter field lets sync distinguish generated project-local skills from hand-written ones. But two failure modes: (a) user copy-pastes a materialized file to start a hand-written skill, forgetting to remove the marker → next sync wipes their edits; (b) user hand-edits a materialized file directly → next sync clobbers changes with no warning.

**Mitigation**: Add a body-level comment to materialized files: `<!-- Generated by bin/build-config. Do not edit — edit .build/config.yaml and re-sync. -->` (immediately after frontmatter, so it appears in editor previews). Before overwriting any managed file, sync does a whole-file body compare against what it would regenerate right now. Bodies match → safe to overwrite. Bodies differ → sync warns and skips, requires `--force` to proceed. No persisted hash needed (simpler than the original SHA256 proposal, same protection).

**Verification**: (a) Copy a materialized file to `hand-written.md`, remove `managed:` field, run sync — hand-written.md is untouched. (b) Hand-edit a materialized file's body, run sync — sync prints warning and skips, exits non-zero.

## Risk 5: Bucket assignment defaults are contested

**Problem**: Plan proposes assigning `/ship` to `execution` and `/wrap` to `mechanical`. But `/ship` internally runs pre-mortem, review, PRD generation, PRD execution, and merge — it spans all three buckets. Any single-bucket assignment is a compromise, and the "best" model varies by phase. Users may find the default surprising and blame the config system, not the assignment.

**Mitigation**: Two-part. (a) Add explicit `# Bucket rationale` comments in `config.yaml.example` explaining why each bucket picked what it did, and pointing out that per-skill overrides win — nudging the user toward "you can override this." (b) In the `bin/build-config show` output, show source-layer breakdown ("skill: ship → opus | source: task_models.execution → sonnet, but overridden by global.skills.ship → opus") so users can trace surprising values.

**Verification**: `config.yaml.example` includes rationale comments for each bucket. `bin/build-config show ship` prints the resolution trace (not just the final value).

## Risk 6: Verification steps 5, 8, 9 need a scratch project + live session

**Problem**: Plan's verification steps 5–9 require operating outside the build-os source repo (scratch project) and inside a live Claude Code session (step 8: confirm frontmatter model wins for the turn). These are not achievable inside the `/build` autonomous execution — orchestrator can't spin up a Claude Code session or simulate `/model` switching mid-turn.

**Mitigation**: Split verification into two tiers in the PRD: (a) **automated** — steps 1–4, 7 run inside the worktree by orchestrator (config init, sync, grep, `show`). (b) **manual wrap** — steps 5, 6, 8, 9 land as `/wrap` checklist items with explicit "run this yourself" framing. The ship report surfaces the manual list.

**Verification**: PRD tasks include a "manual verification checklist" task at the end whose deliverable is a bulleted checklist in `.build/verification-manual.md`, not code.

## Risk 7: install.sh & bin/new-project changes are cross-cutting

**Problem**: Editing `install.sh` and `bin/new-project` in the same PRD as the new `bin/build-config` CLI creates dependency ordering: `bin/build-config` must exist before `install.sh` prints the hint about it; `config.yaml.example` must exist before `bin/new-project` copies from it. If tasks execute out of order, later tasks fail with "file not found."

**Mitigation**: PRD task order enforces: (1) create `bin/build-config`, (2) create `build/config.yaml.example`, (3) add `bucket:` to `commands/*.md`, (4) edit `build/AGENTS.md`, (5) edit `.gitignore`, (6) edit `install.sh` (references bin/build-config), (7) edit `bin/new-project` (references config.yaml.example). Each task lists explicit `depends_on` in the PRD.

**Verification**: PRD task list linearly orders these seven groups; each downstream task has its dependency named.
